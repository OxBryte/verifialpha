# VerifiAlpha

**Provably unfakeable AI trading track record, built on 0G.**

---

## The Problem

Every AI trading signal, copy-trade bot, and "my algorithm did 300% this year" claim on the internet is an **unverifiable assertion**. There is no way for an observer to know whether:

- The track record is real or fabricated
- Losing trades were silently deleted
- Decisions were backdated after the outcome was already known
- The published signals match what the AI actually produced

This is not a theoretical concern. The crypto and trading spaces are saturated with influencers and products making performance claims that cannot be independently verified. The entire trust model is "take my word for it."

## The Solution

VerifiAlpha eliminates the possibility of fakery by binding every step of the AI decision pipeline to cryptographic proof, **before the market outcome is known**:

```
Live Price Data
  → Deterministic Technical Features (EMA crossover, RSI)
    → AI Judgment (TEE-verified inference via 0G Compute)
      → Immutable Decision Receipt (0G Storage)
        → On-chain Timestamped Log (0G Chain)
          → Verifiable PnL Track Record
```

The result is an **append-only, ordered, independently verifiable** AI trading track record. You cannot backdate a decision. You cannot delete a losing trade. You cannot claim the AI said something it didn't. Every decision is public, timestamped, and cryptographically bound to the inputs that produced it.

## How It Works — Step by Step

### 1. Deterministic Feature Computation

The AI does **not** do the math. A deterministic TypeScript module computes quantitative trading features from live OHLCV price data:

- **EMA Crossover** (12/26 period) — identifies trend direction and momentum shifts
- **RSI** (14 period) — measures overbought/oversold conditions
- **Trend Classification** — combines EMA and RSI into a regime label (bullish momentum, bearish momentum, overbought, oversold, sideways)

This separation is deliberate. LLMs are unreliable at arithmetic but strong at reasoning and judgment. By pre-computing the features with exact, reproducible math, we ensure the inputs to the AI are correct and auditable. The same candle data will always produce the same features.

```typescript
interface Features {
  emaFast: number;      // 12-period EMA
  emaSlow: number;      // 26-period EMA
  rsi: number;          // 14-period RSI
  emaCrossover: "BULLISH" | "BEARISH" | "NEUTRAL";
  trend: string;        // human-readable regime
}
```

### 2. TEE-Verified AI Inference (0G Compute)

The computed features are injected into a structured prompt and sent to an AI model running inside a **Trusted Execution Environment (TEE)** via 0G Compute Network.

The prompt constrains the model to return a strict JSON response:

```
You are a disciplined trading analyst. You are given precomputed technical
features for BTC-USD. Do NOT recompute or invent numbers. Based ONLY on
these features, return a single trading decision.

Features: {"emaFast":63336.02,"emaSlow":63914.02,"rsi":40.94,...}

Respond with ONLY this JSON, no prose:
{"decision":"LONG"|"SHORT"|"FLAT","confidence":<0..1>,"rationale":"<=240 chars"}
```

The TEE attestation via `processResponse()` proves that:
- The response was generated by the claimed model
- Inside a trusted execution environment
- The output has not been tampered with

This is what makes the AI's judgment **verifiable** — it's not just an API call to a black box. The TEE binds the model, the input, and the output into an attested computation.

### 3. Decision Receipt Assembly

After inference, a `DecisionReceipt` is assembled containing everything needed to independently verify the decision:

```typescript
{
  id: "2acd3859-...",
  agentId: "verifialpha-v1",
  asset: "BTC-USD",
  timestamp: 1781907300873,
  priceAtDecision: 62959,

  // The exact features the AI saw
  features: { emaFast: 63336.02, emaSlow: 63914.02, rsi: 40.94, ... },

  // Binds the prompt to the attested output
  promptHash: "8904676cae48ca2766b353d36eb5cd68...",
  model: "qwen/qwen2.5-omni-7b",
  providerAddress: "0xa48f01287233509FD694a22Bf840225062E67836",
  chatID: "chatcmpl-dbc783f2-...",

  // The AI's judgment
  decision: "SHORT",
  confidence: 0.85,
  rationale: "Bearish EMA crossover and bearish momentum indicate selling opportunity.",

  // Proof chain
  teeValid: true,
  storageRootHash: "0x7740722f42ec0a3f...",
  storageTxHash: "0xc080d0adeabd19eb...",
  chainDecisionId: 0,
  chainTxHash: "0x2706375beedfd089..."
}
```

### 4. Immutable Storage (0G Storage)

The complete receipt JSON is uploaded to **0G Storage** as in-memory data using `MemData`. The SDK computes a Merkle tree over the data and returns:

- **Root Hash** — a content-addressable identifier. The same data will always produce the same hash. Any modification produces a different hash.
- **Transaction Hash** — proof that the upload was committed to the 0G storage network at a specific time.

The receipt is now **immutable**. It cannot be altered after upload without changing the root hash, which would break the on-chain reference.

### 5. On-Chain Timestamped Log (0G Chain)

The storage root hash and the decision action (LONG/SHORT/FLAT) are logged to the `TrackRecord` smart contract on 0G Chain:

```solidity
contract TrackRecord {
    event Decision(
        uint256 indexed id,
        address indexed agent,
        bytes32 storageRoot,
        int8 action,        // 1 = LONG, -1 = SHORT, 0 = FLAT
        uint64 ts
    );

    uint256 public count;
    mapping(uint256 => bytes32) public roots;

    function logDecision(bytes32 storageRoot, int8 action) external returns (uint256 id) {
        id = count++;
        roots[id] = storageRoot;
        emit Decision(id, msg.sender, storageRoot, action, uint64(block.timestamp));
    }
}
```

This is the critical link. The `block.timestamp` is set by the blockchain — not by the agent. Once a decision is logged, its position in the ordered sequence is permanent. The contract is append-only: there is no `deleteDecision` or `updateDecision` function.

**This means:**
- Every decision has a provable ordering (decision #0 came before #1)
- Every decision has a blockchain-certified timestamp
- The decision was committed **before** the market moved to reveal the outcome
- The storage root hash links back to the full, immutable receipt

### 6. Mark-to-Market PnL

On subsequent agent runs, open positions are marked against the current price. The PnL is computed directionally:

- LONG: `(currentPrice - entryPrice) / entryPrice`
- SHORT: `-(currentPrice - entryPrice) / entryPrice`

This produces the equity curve — a running cumulative PnL that represents the agent's verifiable track record. Because every decision that feeds into this curve is on-chain and timestamped, the curve itself is **provably accurate**.

## Architecture

```
                 ┌─────────────────────────────────────────────┐
                 │  AGENT LOOP (server-side, Node.js)           │
                 │                                              │
  Price feed ──► │  1. Fetch OHLCV (CoinGecko, no auth)        │
                 │  2. Compute deterministic features (TS)      │
                 │  3. Build prompt ──► 0G COMPUTE ─────────────┼──► TEE inference
                 │  4. processResponse() ──► isValid            │    (qwen2.5-omni-7b)
                 │  5. Build DecisionReceipt JSON               │
                 │  6. Upload receipt ──► 0G STORAGE ───────────┼──► rootHash + tx
                 │  7. logDecision(rootHash, action) ───────────┼──► 0G CHAIN
                 │  8. Mark-to-market open positions ──► PnL    │    (TrackRecord.sol)
                 │  9. Index everything ──► local DB cache      │
                 └─────────────────────────────────────────────┘
                                      │
                          Next.js dashboard reads DB
                          (equity curve, decision log, verify panel, leaderboard)
```

**Hard rule:** every 0G SDK call runs server-side (Next.js route handlers). The storage SDK imports `fs`/`crypto` at load time and is incompatible with the browser. The browser only reads finished, indexed results from the API.

## 0G Integration — All Three Primitives

This project uses all three core 0G Network primitives. Each serves a distinct, necessary role in the verification chain:

| Primitive | SDK | Purpose | What it proves |
|-----------|-----|---------|---------------|
| **0G Compute** | `@0glabs/0g-serving-broker` | TEE-verified model inference | The AI's decision was produced by the claimed model inside a trusted execution environment |
| **0G Storage** | `@0gfoundation/0g-ts-sdk` | Immutable decision receipt storage | The complete decision data (features, prompt, response, hashes) exists in an unalterable form |
| **0G Chain** | Solidity + ethers v6 | Append-only timestamped decision log | The decision was committed at a specific time, in a specific order, before the outcome was known |

Remove any one of these and the verification chain breaks:
- Without Compute: the AI output could be fabricated
- Without Storage: the receipt could be altered after the fact
- Without Chain: the timestamp could be backdated

## Dashboard

The frontend is a single-page Next.js dashboard with four sections:

1. **Hero + Run Agent Now** — One-click button to trigger a live decision during demos. Shows the thesis in one line.

2. **Equity Curve** — Cumulative paper PnL over time, rendered as an SVG chart. Every point on this curve traces back to a verifiable on-chain decision.

3. **Decision Log** — Most recent decisions first. Each row shows asset, time, decision (LONG/SHORT/FLAT), confidence, and current PnL. Expanding a row reveals the **Verify Panel**:
   - TEE attestation status (valid/invalid)
   - 0G Storage root hash with explorer link
   - 0G Chain transaction hash with explorer link
   - The AI's rationale
   - The exact features the decision was based on
   - Model name and provider address

4. **Leaderboard** — Agents ranked by verifiable PnL. Sets up the framework for multi-agent competition in later rounds.

## Quick Start

### Prerequisites

- Node.js 18+
- A burner wallet private key
- Testnet A0GI tokens (~1.5 A0GI recommended)

### 1. Clone and install

```bash
git clone https://github.com/<your-username>/verifialpha.git
cd verifialpha
npm install --legacy-peer-deps
```

### 2. Configure environment

```bash
cp .env.example .env.local
# Edit .env.local — set your PRIVATE_KEY
```

### 3. Fund your wallet

Get testnet A0GI from [faucet.0g.ai](https://faucet.0g.ai). You need approximately 1.5 A0GI to cover:
- Compute ledger funding (1.0 A0GI minimum provider reserve)
- Storage upload fees
- Chain transaction gas

### 4. Verify integrations

```bash
npm run test:phase0
```

This runs the gate test — confirms 0G Compute inference and 0G Storage upload both work.

### 5. Deploy the smart contract

```bash
npm run deploy:contract
```

Compiles `TrackRecord.sol` with solcjs and deploys to 0G Galileo testnet. The contract address is automatically written to `.env.local`.

### 6. Run the app

```bash
npm run dev
```

Open [http://localhost:3000](http://localhost:3000). Click **"Run Agent Now"** to trigger a live decision.

## Why This Matters

The AI trading space has a fundamental trust problem. Every track record you see online could be:

- **Fabricated** — made up entirely
- **Cherry-picked** — only winning trades shown
- **Backdated** — decisions recorded after the outcome was known
- **Selectively published** — multiple accounts run, only the lucky one is shown

VerifiAlpha makes all of these impossible. Not impractical. Not discouraged. **Impossible.** The cryptographic chain from TEE-attested inference to immutable storage to on-chain timestamping leaves no room for manipulation.

This is not just a hackathon demo. This is the foundation for a world where AI trading performance claims can be independently verified by anyone, without trusting the person making the claim.
